# 3フォーマット比較・変換ルールマッピング仕様書

**作成日:** 2026-04-19  
**対象フォーマット:** Freee P/L CSV / budget-tool CSV / 患者請求CSV

---

## 1. 概要・データフロー

ワイズ株式会社の財務データは以下の3系統のCSVフォーマットを持つ。

```
調剤システム
    │
    ▼
【患者請求CSV】（調剤システム月次出力）
    │ 手動集計（現状）
    ▼
【Freee P/L CSV】（月次推移：損益計算書）
    │ Freee取込ボタン（ブラウザUI）
    ▼
【budget-tool CSV】（Firestore / エクスポート）
```

### 各フォーマットの位置づけ

| フォーマット | 生成元 | 粒度 | 用途 |
|------------|--------|------|------|
| 患者請求CSV | 調剤システム | 患者×月 | 個別請求・保険請求管理 |
| Freee P/L CSV | Freee会計 | 勘定科目×月×部門 | 財務会計（法定） |
| budget-tool CSV | budget-tool (Firestore) | 管理科目×月×部門 | 管理会計・予実管理 |

## 2. フォーマット詳細

### 2.1 Freee P/L CSV（月次推移：損益計算書）

詳細は `docs/freee_csv_mapping.md` を参照。概要のみ示す。

| 項目 | 仕様 |
|------|------|
| 文字コード | Shift-JIS（麗華でUTF-8変換済み） |
| 列数 | 20列（L0〜L5の6階層 + 空白1列 + 12か月 + 累計） |
| 識別子 | 1行目に「月次推移」含む |
| 部門識別 | CSV内に部門名なし → 物件名ヒューリスティック判定 |

**主要列:**
```
Col 0: L0（合計行ラベル）
Col 1: L1（大分類: 売上高/販売管理費/営業外収益...）
Col 2: L2（勘定科目: 技術料/給料手当/車両費...）
Col 3: L3（補助科目: 中村ビル/タイムズカーシェア...）
Col 4: L4（細分類）
Col 5: L5（最細分類）
Col 6: 空白
Col 7〜18: 月別金額（2月=Col7 ... 1月=Col18）
Col 19: 期間累計
```

---

### 2.2 budget-tool CSV（エクスポート形式）

budget-toolの「エクスポート」ボタンで出力されるCSV形式。

| 項目 | 仕様 |
|------|------|
| 文字コード | UTF-8（BOM付き） |
| 列数 | 14列（科目名 + 年間合計 + 12か月） |
| ヘッダー | `"科目","年間合計","2026-02","2026-03",...,"2027-01"` |
| 金額単位 | 円（整数） |
| 空値 | 空文字列（0でなく空） |

**フォーマット例:**
```csv
"科目","年間合計","2026-02","2026-03","2026-04",...,"2027-01"
"技術料(KGI)",59500000,28500000,31000000,0,...,0
"薬剤料",12000000,6000000,6000000,0,...,0
"給料手当",36000000,3000000,3000000,...,3000000
```

**全体出力時のセクション区切り:**
```csv
【予算】
"科目","年間合計","2026-02",...
（予算行）

【実績】
"科目","年間合計","2026-02",...
（実績行）

【差異】
"科目","年間合計","2026-02",...
（差異行: 実績-予算）
```

**インポートルール:**
- `【...】`で始まる行: セクション区切り（スキップ）
- `"科目"` または `科目`で始まる行: ヘッダー（スキップ）
- 科目名は`findItemByName()`で正規化マッチング（前後スペース・全角スペース除去、「・」プレフィックス補完）

### 2.3 患者請求CSV（調剤システム出力）

調剤薬局システムから月次出力される患者別請求データ。
サンプルファイル: `AI_secretary_reika/received_files/20260417________.txt`（UTF-8変換済み、元Shift-JIS）

| 項目 | 仕様 |
|------|------|
| 文字コード | Shift-JIS（麗華でUTF-8変換済み） |
| 区切り文字 | カンマ（,） |
| 引用符 | なし（一部フィールドに全角スペース含む） |
| 行構成 | ヘッダー行 + 単位行 + データ行（患者1行） |
| ソート | 五十音順（あいうえお順） |

**列定義（26列）:**

| Col# | ヘッダー名 | 単位 | 例（相川春江） | 備考 |
|------|-----------|------|--------------|------|
| 0 | 患者名 | | 相川　春江 | 全角スペース区切り（姓　名） |
| 1 | (blank) | | | 空欄（フリガナは後述Col23に出現） |
| 2 | 給 | | 0 | 給付割合区分（0=公費全額、7=7割、8=8割、9=9割） |
| 3 | 保険種 | | 公+介 | 保険種別コード（例: 国老+介、国本+介、国高公+介） |
| 4 | 回数 | 回 | 2 | 当月の調剤回数 |
| 5 | 定負担 | 円 | 0 | 患者一部負担金（定額負担） |
| 6 | (介護負担) | 円 | 0 | 介護保険自己負担額 |
| 7 | 選定療養 | 円 | 0 | 選定療養費（医療機関側） |
| 8 | 自費薬 | 円 | 0 | 自費薬剤費（保険外） |
| 9 | 請求額 | 円 | 0 | 患者への合計請求額（未収含む） |
| 10 | 入金額 | 円 | 0 | 当月入金済み額 |
| 11 | 現未収 | 円 | 0 | 現在の未収残高 |
| 12 | 保険計 | 円 | 22240 | 保険者への請求総額（保険点数×10） |
| 13 | 基本点 | 点 | 340 | 基本調剤料（技術料の主要構成要素） |
| 14 | 時加算 | 点 | 0 | 夜間・休日等加算 |
| 15 | 調剤点 | 点 | 96 | 調剤料 |
| 16 | 調加算 | 点 | 0 | 調剤加算 |
| 17 | 指導点 | 点 | 1568 | 薬剤服用歴管理指導料等 |
| 18 | 内服薬 | 点 | 220 | 内服薬剤料（薬剤料の構成要素） |
| 19 | 内服外 | 点 | 0 | 外用薬・注射等薬剤料 |
| 20 | 材料点 | 点 | 0 | 特定保険医療材料料 |
| 21 | 患者ID | | | 患者識別番号（空欄の場合あり） |
| 22 | 分類 | | ボンノールガーデン | 施設名またはチーム名（在宅チーム壱等） |
| 23 | (blank header) | | ｱｲｶﾜ ﾊﾙｴ | **患者カナ名**（ヘッダーは空欄だが実データあり） |
| 24 | 選定療養 金額 | 円 | | 選定療養費の内訳金額 |
| 25 | 内訳 税(10%) | 円 | | 消費税（選定療養費の10%） |

> **注意**:
> - Col 23 のヘッダーが空欄だが、実データには患者カナ名（半角カナ）が入る
> - Col 21（患者ID）が空欄の場合がある。patient-dbとの突合は患者名+カナで行う必要がある
> - 保険計（Col 12）= (全点数合計) × 10 の関係（1点=10円）

## 3. 3フォーマット間の変換ルールマッピング表

### 3.1 患者請求CSV → Freee P/L CSV（売上項目）

> この変換は現状**手動**で行われる（税理士が月次Freeeに入力）。
> 将来的に自動化する際の参考変換ルール。

| budget-tool ID | Freee L2科目 | 患者請求CSVでの算出方法 | 算出の確実性 |
|---------------|-------------|----------------------|------------|
| gijutsu | 技術料 | Σ(基本点+時加算+調剤点+調加算+指導点) × 10 | **参考値**（Freee入力値と若干差異あり） |
| yakuzai | 薬剤料 | Σ(内服薬+内服外) × 10 | **参考値** |
| chiryo | 治療器材料 | Σ(材料点) × 10 | **参考値** |
| - | 売上高合計 | Σ(保険計) + Σ(定負担) + Σ(介護負担) + Σ(自費薬) | 患者請求額の合計と一致 |
| - | (保険請求) | Σ(保険計) | 保険者への請求額（売上の主体） |
| - | (患者一部負担) | Σ(定負担) + Σ(介護負担) | |
| - | 自費薬 | Σ(自費薬) | uriage_otherに吸収 |

> **算出の限界**: 点数→円換算は点数×10だが、加算の計算方法は月によって変わる。
> Freee入力値が正式値であり、患者請求CSVは**検算・照合**に使うべきもの。

**保険種別コードの解読:**

| 保険種コード | 主な保険 | 給付割合(Col2) |
|-----------|---------|--------------|
| 国老+介 | 国保後期高齢者＋介護 | 9（9割給付） |
| 国本+介 | 国保一般＋介護 | 7〜8 |
| 国高公+介 | 国保高齢者＋公費＋介護 | 8〜9 |
| 公+介 | 公費（生活保護等）＋介護 | 0（全額公費） |

## 4. patient-db CSV（患者基本情報）との関係

patient-db は患者**基本情報**（デモグラフィクス）を管理する。患者請求CSVとは別システム。

### 4.1 patient-db エクスポートCSVフォーマット

| Col# | ヘッダー | Firestoreフィールド | 患者請求CSVとの対応 |
|------|--------|------------------|------------------|
| 0 | 患者ID | patientId | 患者請求CSV Col 21（空欄の場合あり） |
| 1 | 氏名 | name | 患者請求CSV Col 0（患者名） |
| 2 | カナ | kana | 患者請求CSV Col 23（blank header） |
| 3 | 生年月日 | birthday | 患者請求CSVにはなし |
| 4 | 性別 | gender | 患者請求CSVにはなし |
| 5 | 郵便番号 | zip | 患者請求CSVにはなし |
| 6 | 住所 | address | 患者請求CSVにはなし |
| 7 | 電話番号 | phone | 患者請求CSVにはなし |
| 8 | 担当店舗 | store | 患者請求CSV Col 22（分類/施設名）から推定可能 |
| 9 | 備考 | note | 患者請求CSVにはなし |

### 4.2 患者請求CSV → patient-db の突合ルール

患者請求CSVをpatient-dbに連携する場合、突合キーは以下の優先順で使用：

```
1次: 患者ID（Col 21）が存在する場合 → patientIdで一致
2次: 患者名（Col 0）+ カナ（Col 23）の組み合わせで名寄せ
3次: 患者名のみでの部分一致（同姓同名リスクあり → 要確認）
```

> **注意**: 現状、患者請求CSVへのpatient-db自動取り込みは未実装。
> 設計確定後に実装予定。詳細は `project_patient_db_csv.md` 参照。

## 5. フォーマット差分サマリー

| 比較軸 | Freee P/L CSV | budget-tool CSV | 患者請求CSV |
|-------|--------------|----------------|-----------|
| 文字コード | Shift-JIS→UTF-8 | UTF-8（BOM付き） | Shift-JIS→UTF-8 |
| 列構造 | 6階層+月別 | 科目名+月別 | 患者情報+点数 |
| 時系列粒度 | 月次（年12か月） | 月次（年12か月） | 月次（患者×月） |

| 金額単位 | 円（整数） | 円（整数） | 円または点（×10換算） |
| 科目体系 | 法定勘定科目 | 独自管理科目 | 保険点数区分 |
| 部門判定 | 物件名ヒューリスティック | 明示的（ユーザー選択） | 分類列から推定可 |
| 自動化状況 | **部分自動**（UIから取込） | **手動**（UI操作必要） | **手動**（目視確認） |

## 6. 変換チェーンの現状と自動化ギャップ

```
【現状フロー】
患者請求CSV → 手動確認 → Freee手入力 → Freee P/L CSV → 手動UIアップロード → budget-tool

【理想フロー】
患者請求CSV → 自動集計スクリプト → Freee API OR CSV → 自動budget-tool反映
```

| ステップ | 現状 | 自動化難度 | 主な障壁 |
|--------|------|----------|---------|
| 患者請求CSV → Freee | 税理士が手動入力 | 高 | 税理士の作業フローに依存 |
| Freee CSV 出力 | 税理士が手動出力・送信 | 中 | Freee API連携で省略可能 |
| CSV受信・変換 | 麗華が自動受信・UTF-8変換 | ✅ 実装済み | |
| budget-tool取込 | UIから手動アップロード | 中 | Node.js→Firestore直書きで省略可能 |
| 部門判定 | ヒューリスティック自動判定 | ✅ 実装済み | 信頼度確認は手動必要 |
